# Task Status Header Chip Design

## 背景

当前轻量任务状态使用内容区右上角浮层展示。详情面板打开时，浮层会覆盖面板标题栏和关闭按钮，导致全局状态提示侵占局部操作区。任务状态本质上是全局系统状态，应该进入全局 header，而不是悬浮在文件内容和详情面板上方。

## 目标

- 将“正在处理任务”从内容区浮层改为 header 内的任务状态 chip。
- 重新整理 header 的旧 chip 和入口位置，让主导航、全局状态、工具入口各自稳定。
- 任务状态可被持续感知，但不能遮挡详情关闭按钮、图片操作按钮、底部批量操作条等关键操作。
- 点击任务状态 chip 能展开小面板查看更多信息，并提供进入任务队列的入口。

## Header 布局

Header 分为三段：

- 左侧：Tidy logo 和品牌入口。
- 中间偏左：主导航 segmented control，包含“全部文件 / 文件夹/归档 / 回收站”。
- 右侧：全局状态与工具入口，顺序为“任务状态 chip / 工具 / 系统管理”。

推荐结构：

```text
[Tidy]   [全部文件 | 文件夹/归档 | 回收站]                 [处理中 37%] [工具] [系统管理]
```

选择右侧工具区第一位的原因：

- 主导航表示当前视图，不应被任务状态打断。
- 任务状态是全局系统状态，属于工具区，但优先级高于工具箱和系统管理。
- 系统管理是任务详情的落点，任务 chip 靠近系统管理，符合空间记忆。
- 右侧内容区不再出现任务浮层，详情面板的关闭按钮不再被覆盖。

## Task Chip 默认形态

默认 chip 保持紧凑，避免撑乱 header：

- 高度与 header 中其他按钮一致。
- 左侧使用旋转的 `RefreshCw` 图标表示运行中。
- 主文案显示短状态，例如“处理中”。
- 进度优先显示百分比；没有可计算百分比时显示 `processed / total`；没有进度时只显示状态。
- `type · mode` 不放在主 chip，放到 hover title 或展开面板。
- 可选使用底部 2px 进度线表达推进，不使用大阴影 toast 风格。

示例：

```text
[↻ 处理中 37%]
```

或：

```text
[↻ 11800 / 31802]
```

## 点击展开面板

点击 chip 后在 header 下方打开 popover，`align="end"`，展示任务细节和入口：

```text
正在处理任务
enrich · missing
11800 / 31802
[progress bar]

查看任务队列
```

行为要求：

- 展开面板挂在 chip 下方，不进入内容区右上角。
- 点击“查看任务队列”进入系统管理中的任务队列区域。
- 面板只在用户主动点击时出现；默认不遮挡详情面板顶部操作。

## 状态

- `running`：蓝色旋转图标，文案“处理中”。
- `queued`：中性或琥珀色等待图标，文案“等待中”。
- `failed` / `interrupted`：红色 `AlertTriangle`，文案“任务失败”，展开面板展示错误摘要和查看任务队列入口。
- 多个活跃任务：主 chip 显示聚合状态，例如“2 项处理中”；展开面板列出任务。
- 无活跃任务且最新任务未失败时不显示 chip。

## 响应式降级

普通宽度：

```text
[Tidy] [主导航]                         [处理中 37%] [工具] [系统管理]
```

窄宽度：

```text
[Tidy] [主导航]                         [↻ 37%] [工具] [管理]
```

更窄时，任务 chip 可以只保留图标并通过 title 或 popover 展示详情，但仍保留在 header 右侧工具区，不退回内容区右上浮层。

## 不做

- 不再默认使用内容区右上角浮层。
- 不把任务 chip 放进主导航 segmented control。
- 不把拖动作为默认方案。拖动只适合作为后续增强，因为它需要记忆位置、边界吸附和窗口变化处理，而且会把布局冲突交给用户修补。

## 验收标准

- 打开详情面板时，任务状态不会遮挡详情标题、关闭按钮或预览图操作。
- Header 中主导航、任务状态、工具和系统管理的视觉层级清晰。
- 任务运行、排队、失败和无任务状态都有明确展示。
- 点击任务 chip 能看到 `type · mode`、进度和进入任务队列的入口。
- 窄宽度下 header 不出现文字挤压、重叠或按钮溢出。
