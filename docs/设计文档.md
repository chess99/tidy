# Tidy 设计文档

> 同步阅读：[`docs/决策记录.md`](决策记录.md)（记录长期稳定的定位/取舍/一致性策略；本文更偏“架构与数据流/接口约定”）。

## 1. 目标与非目标

### 目标

- **整理/去重**：把“相同内容”的重复副本识别出来，并在整理时执行“保留一份、其余副本直接物理删除”（回收站仅用于用户删除的最后留底）。
- **真实落盘**：整理/删除动作直接对文件系统生效（移动/入 Trash），而不是虚拟归类。
- **可恢复一致性**：通过 `file_ops` 日志与 `/sync` 对账，处理中断/崩溃后仍可恢复到一致状态。
- **大库可用**：扫描与 hash/缩略图补全是渐进的，前端通过 SSE 增量更新，不做整列表刷新。

### 非目标（至少当前阶段）

- 不做“相册式”人像/地点浏览体系（可以预留标签与未来模型扩展位）。
- 不追求云端/多端同步；优先保证本地落盘与 DB 的一致性。

## 2. 系统架构（Local Server + SPA）

- **Backend**：Node.js + Express
- **DB**：SQLite（`better-sqlite3`）
- **Scanner**：walk → 先写 `files`（可见性）→ 队列异步 enrich（hash/metadata/thumb）
- **Thumbnail**：best-effort（优先 image/*；RAW 如 DNG/CR2/NEF/ARW 等在 sharp 不支持时改为提取内嵌 JPEG 预览再生成），并用 `thumb_updated_at` 做 cache busting
- **Frontend**：React + TanStack Query + 虚拟列表（`FilesGrid`）
- **增量更新**：SSE（`/api/changes/stream`）+ batch 拉取 + react-query cache patch

## 3. 产品视图（三 Tab）

### Tab1：全部文件（All Files）

- **以 `files(path)` 为数据源**：允许重复、允许无 hash；排除已删除的文件（`assets.status='trash'`）。
- **卡片直接可见**：未整理显示文件名+路径；已整理显示 `organized_to`；重复显示 `dup_count`。
- **筛选/批量操作**：左侧常驻筛选栏（范围/状态/日期/路径/hash），筛选后多选 → “添加到(文件夹/归档)”或“入 Trash”。
- **详情面板**：点击文件/资产后展示 hash、拍摄时间、相机信息、所有物理路径等，并支持“一键写入筛选”（chips）。

### Tab2：文件夹/归档（Folders/Archives）

- **以 `albums` 为数据源**：展示真实归档文件夹列表（对应 `<MANAGED_ROOT>/<albumName>`）。
- 点击进入后展示该文件夹里的内容（按 hash 去重的资产列表）。

### Tab3：回收站（Trash）

- **以 `assets.status='trash'` 为数据源**：用于浏览“已删除”的内容。
- **落盘语义**：回收站只保留该内容的**最后一份物理副本**（位于配置的 `TRASH_DIR`），其余副本会被物理删除。

### 实用工具：检查重复项（不占 Tabs）

- 入口：主界面右上角扳手（Popover）。
- 能力：按 **hash 完全重复** 与 **pHash 相似重复** 分组展示，逐组选择“保留/删除副本”。
  - 当某 hash 的所有物理实例都被选择删除时，升级为删除该 asset（保留最后一份到回收站）。

## 4. 目录/配置（扫描根与托管目录）

- **扫描根（scanRoots）**：用户自行添加的扫描目录列表（config.json），无默认值；Discover 仅扫描已启用的根。
- **MANAGED_ROOT**：用户选择的托管归档目录（整理目标）；默认 `~/Pictures/Tidy`。在 **设置 → 整理与回收站目录** 中配置，持久化于 config.json。
- **TRASH_DIR**：工具回收站（默认 `DATA_DIR/trash`），用于用户删除内容的最后一份留底。同上，在设置页配置。
- **扫描边界**：扫描根可任意指定；若包含 `MANAGED_ROOT` 或 `TRASH_DIR`，Enrich 会按路径区域把其中文件回填为 `sorted/trash`，用于 DB 丢失/损坏或人工整理后的恢复。

> 相关实现集中在 `server/src/config.js`、`server/src/configStore.js`、`server/src/jobs/handlers/discover.js`（扫描）与 `server/src/jobs/handlers/enrich.js`（按路径区域重建状态）。

## 5. 数据模型（SQLite）

### 5.1 `files`（物理文件实例 / Where）

- `id`（PK）
- `path`（unique）
- `hash`（nullable）
- `size` / `mtime_ms` / `ext` / `mime_guess`
- `hash_status`：`pending|done|error`
- `thumb_status`：`pending|ready|unsupported|error`
- `thumb_updated_at`：缩略图版本戳
- `phash`：图片感知哈希（pHash, 64-bit hex，用于相似重复检测）
- `phash_status`：`pending|done|error|unsupported`
- `discovered_at` / `updated_at` / `scanned_at`
- `missing`：缺失标记（同步/删除/外部移动后可由维护工具标记）

### 5.2 `assets`（内容资产 / What）

- `hash`（PK）
- `mime_type` / `size`
- `metadata`（JSON）
- `taken_at`（排序时间：优先 EXIF，其次 mtime 等兜底）
- `status`：`inbox|sorted|trash|ignored`
- `target_path`：落盘整理后的目标路径（用于展示/对账）
- `updated_at` / `thumb_updated_at`

### 5.3 `albums` / `album_assets`（归档文件夹 / 按 hash 归档）

- `albums(id,name,created_at,updated_at)`
- `album_assets(album_id,hash,added_at)`：成员以 hash 记录，保证“去重与永久生效”。

### 5.4 `file_ops`（落盘操作日志 / 一致性与恢复）

- 记录 `move/trash/delete` 操作：`op, hash, file_id, from_path, to_path, album_id, status(pending/done/error), error, created_at, updated_at`
- `/sync` 会重放 `pending` 操作，并处理历史遗留 trash 标记的对账。

### 5.5 `tags` / `asset_tags`（可扩展的手动标签）

- 标签绑定在 `assets(hash)` 上，避免重复副本重复标注。
- 类型支持 `place|subject|person|food|other`（后续可扩展筛选/模型自动打标）。

### 5.6 `changes`（增量更新日志）

- `id` 自增游标（cursor）
- `entity`/`entity_id`（file/asset/tag…）
- `type`（upsert/delete/thumb_ready/…）
- `ts`

## 6. 核心工作流

### 6.1 扫描（Discover → Enrich）

- **Discover**：walk 时立刻 upsert `files`（可见性）+ 写 `changes(file, upsert)`；再把任务推入队列。
- **Enrich**：队列异步计算 hash → upsert `assets` → 回写 `files.hash/hash_status` → best-effort 缩略图 → 写 `changes`。
  - 同时基于文件路径区域重建 `assets.status/target_path` 与 `album_assets`（`MANAGED_ROOT` = sorted，`TRASH_DIR` = trash）。

### 6.2 整理（Add to /organize）

- 输入：hashes + album（id 或 name）
- 规则：同 hash 多份物理文件 → 选一份 move 到 `<MANAGED_ROOT>/<album>`；其余副本 **物理删除**（不进入回收站）。
- 一致性：每个 move/trash 写 `file_ops`，并在 DB 更新后标记 done；崩溃可由 `/sync` 对账恢复。

### 6.3 删除（入 Trash）

- 以 hash 为单位，批量标记 `assets.status='trash'`。
  - **仅保留最后一份副本** move 到 `TRASH_DIR` 并保留 `files` 行（用于展示/恢复）。
  - 其余副本 **物理删除**（`file_ops(delete)`）。
  - `/sync` 可兜底重放 pending ops 并对齐到上述语义。

### 6.4 同步/对账（任务：`sync`）

- 通过创建后台任务 `type=sync` 触发（`POST /api/jobs`）
- 重放 `file_ops.status='pending'`（崩溃恢复）
- 处理历史遗留 trash 标记，转成可追踪的 `file_ops` 并落盘执行

## 7. API 约定（当前实现）

### 7.1 任务触发（统一入口）

- `POST /api/jobs`：创建后台任务（discover/enrich/thumbs_rebuild/faces_scan/clip_enrich/clip_index/sync 等）
- `GET /api/jobs`：查看任务队列

### 7.2 Files（Tab1）

- `GET /api/files`：分页 files（LEFT JOIN assets），支持筛选：
  - `filter=all|media|camera`
  - `organized=0|1`
  - `hasDup=0|1`
  - `from/to`（ms）
  - `pathContains`（LIKE）
  - `hash`（精确）
  - `similarKind=phash|clip`（相似检索：pHash 或 CLIP）
  - `similarToFileId=<file_id>`（seed：物理文件 id）
  - `similarThreshold=0..32`（仅 pHash：汉明距离阈值）
  - `similarTopK=1..5000`（仅 CLIP：近邻数量）
  - `similarMinScore=<number>`（仅 CLIP：cosine similarity 最低门槛，越大越严格）
- `POST /api/files`：智能搜索（text→image），按相似度排序返回 files（含 `score`），并与上述筛选条件叠加：
  - `smartQuery=<text>`
  - `smartTopK=1..5000`
  - `smartMinScore=<number>`
- `GET /api/files/date-index`：返回 month/day 分界点（必须与 `/files` 同筛选语义）
- `GET /api/files/batch?ids=...`：批量取最新 files（用于 SSE patch）

### 7.2.1 智能搜索（CLIP）

- `POST /api/files`：文本搜图（text→image），按相似度排序返回 files（含 `score`），并支持叠加其它筛选条件。

### 7.3 Assets（详情面板 + 批量状态）

- `GET /api/assets/:hash`：资产详情（含该 hash 的所有 files）
- `GET /api/assets/batch?hashes=...`：批量取最新 assets
- `PATCH /api/assets/:hash`：更新状态
- `POST /api/assets/batch-status`：批量更新状态（如批量 trash）
- `GET /api/assets/:hash/thumb`：缩略图（用 `?v=` 版本参数刷新）

### 7.4 Albums / Organize（Tab2）

- `GET /api/albums`：列出归档文件夹
- `POST /api/albums`：创建归档文件夹
- `GET /api/albums/:id/assets`：列出该归档内的资产
- `POST /api/organize`：落盘整理 + 去重

### 7.5 Duplicates（实用工具：检查重复项）

- `GET /api/duplicates/groups`：按 `kind=hash|phash` 返回重复项分组（`phash` 支持 `threshold`）。
- `POST /api/duplicates/apply`：应用当前组的“保留/删除副本”决策（必要时升级为 asset 删除并保留最后一份到回收站）。

### 7.6 Tags（扩展位）

- `GET /api/tags?type=` / `POST /api/tags`
- `GET /api/tags/asset/:hash`
- `POST /api/tags/asset/:hash` / `DELETE /api/tags/asset/:hash/:tagId`

### 7.7 变更流（SSE）

- `GET /api/changes/stream?cursor=`：推送 `{cursor, files[], assets[]}`

## 8. 维护与脏数据处理（概念）

- **大小写路径重复**：不强制全局归一；提供 `repair-db` 工具报告/可选修复。
- **删库重扫**：hash 必然重算；缩略图应尽量复用（若 `<thumbDir>/<hash>.jpg` 已存在则跳过生成并回填 DB）。
