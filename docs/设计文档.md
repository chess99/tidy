# Tidy 设计文档

## 1. 背景与核心目标

这是一个**本地文件/相册管理工具**（Node.js + SQLite），核心诉求是：

- **全量管理**：不仅管理照片，也管理同目录下的其他文件类型（视频、RAW、文本、杂项等）。
- **抗“复活”**：对文件做“逻辑删除/整理”等操作后，即便文件被同步软件/误操作再次复制回目录，也要能基于内容识别并保持“永久生效”的逻辑状态。
- **高可用体验**：扫描大目录（数万文件）时，前端应尽早可见，并且后续 hash/缩略图/元数据补全应**增量更新**，避免整列表刷新。

## 2. 产品交互理念（两 Tab）

参考成熟相册 App 的主流程，形成两个视图：

### Tab1：全量文件（All Files）

- 数据源按 **`files`（路径实例）** 展示：允许重复、允许无 hash。
- 扫描过程中：文件被发现后即可出现在列表里；未完成 hash 的条目使用“占位卡片”（大号后缀 + 小号文件名）。
- 点击某条文件：若已完成 hash，可打开其对应的资产详情（`assets(hash)`）。

### Tab2：内容资产库（Library）

- 数据源按 **`assets`（hash 去重）** 展示：一份内容只显示一次。
- 后续整理动作（删除、加入相册等）**以 hash 为主键**，保证“永久生效”。

## 3. 核心数据模型（SQLite）

我们使用 `better-sqlite3`。

### 3.1 表：`files`（物理文件实例，Where）

每条记录代表磁盘上的一个文件路径：

- `id`：自增主键
- `path`：绝对路径（唯一）
- `hash`：内容哈希（可为空，待补全）
- `size` / `mtime_ms` / `ext` / `mime_guess`：用于 Tab1 立即展示与后续处理
- `hash_status`：`pending|done|error`（hash 是否已补全）
- `thumb_status`：`pending|ready|unsupported|error`（缩略图状态）
- `thumb_updated_at`：缩略图版本戳（用于前端 cache busting）
- `discovered_at` / `updated_at` / `scanned_at`：发现/更新/处理时间戳
- `missing`：文件是否缺失（同步/删除后可标记）

### 3.2 表：`assets`（内容资产，What）

每条记录代表“内容”本身（以 hash 去重）：

- `hash`：主键
- `mime_type` / `size`
- `metadata`：JSON（图片类可写 EXIF，非图片可为空或 `{}`）
- `taken_at`：排序时间（优先 EXIF，其次文件 mtime）
- `status`：`inbox|sorted|trash|ignored`（逻辑状态）
- `updated_at`：资产更新时间
- `thumb_updated_at`：缩略图更新时间（前端 `?v=` 用）

### 3.3 表：`changes`（变更日志，用于增量更新）

为了让前端只更新变更项：

- `id`：自增游标（cursor）
- `entity`：`file|asset`
- `entity_id`：`file.id` 或 `asset.hash`
- `type`：如 `upsert|thumb_ready|...`
- `ts`：时间戳

### 3.4 表：`albums` / `album_assets`（相册，按 hash 归档）

相册成员以 `asset.hash` 存储，确保整理动作与“永久生效”一致。

## 4. 扫描与处理管线（Discover -> Enrich）

扫描被拆成两段，以满足“先可见、后补全”：

### 4.1 Discover（遍历发现）

目录遍历遇到文件时：

- 立即 **upsert `files`**（写入 `size/mtime/ext/mime_guess/...`，`hash_status='pending'`）
- 记录一条 `changes(file, fileId, 'upsert')`
- 将任务推入队列（hash/元数据/缩略图在后台异步补全）

### 4.2 Enrich（后台补全）

队列 worker 对每个文件执行：

- 计算内容 hash
- upsert `assets(hash)`（写 `taken_at/metadata/updated_at`）
- 更新 `files.hash/hash_status/...`
- 对可生成缩略图的类型做 **best-effort**：成功则写 `thumb_updated_at` 并记录 `thumb_ready` 变更；失败则标 `unsupported`

> 注：前期缩略图以 `image/*` 为主；RAW/视频等可逐步扩展为 best-effort（例如接入 ffmpeg）。

## 5. 增量更新策略（SSE + Batch + Cache Patch）

### 5.1 为什么不用“整列表刷新”

当库规模达到几万条时，全量刷新会导致：

- 网络与 DB 压力无谓放大
- 虚拟列表滚动位置/已加载页被打断
- 缩略图 404 缓存/重试行为不可控

### 5.2 方案

- 后端每次重要更新写入 `changes`
- 客户端用 **SSE** 订阅 `/api/changes/stream?cursor=...`
- 收到 `{cursor, files:[id], assets:[hash]}` 后：
  - 调用 `/api/files/batch`、`/api/assets/batch` 批量拉取最新行
  - 使用 React Query 的 `setQueriesData` **只 patch 已加载页面中的对应 item**

### 5.3 缩略图更新

缩略图 URL 增加版本参数，保证生成后能强制刷新：

- `.../thumb?v=thumb_updated_at|updated_at`

## 6. API 约定（当前实现）

### 6.1 扫描

- `POST /api/scan`：触发扫描
- `GET /api/scan/status`：扫描进度

### 6.2 Tab1：files

- `GET /api/files?page=&limit=`：分页返回 files（LEFT JOIN assets）
- `GET /api/files/batch?ids=1,2,3`：批量取最新 files

### 6.3 Tab2：assets

- `GET /api/assets?page=&limit=`：分页 assets
- `GET /api/assets/:hash`：资产详情（含路径列表）
- `GET /api/assets/batch?hashes=a,b,c`：批量取最新 assets
- `PATCH /api/assets/:hash`：更新状态（trash/sorted/inbox/ignored）
- `GET /api/assets/:hash/thumb`：缩略图（不存在时返回 404）

### 6.4 变更流

- `GET /api/changes/stream?cursor=`：SSE 变更推送

## 7. 工程/实现层决策记录

- **fastq 队列等待**：`fastq@1.x` 的 `drain()` 不可 `await`，需要使用 `drained()` 等待队列清空，否则会出现“任务 push 全部完成但 worker 未执行完就结束”的现象。
- **并发控制**：队列并发以常量控制（例如 4），后续可配置化；需要在 IO/CPU/DB 负载间平衡。
- **状态落库**：所有关键状态（hash/缩略图/逻辑状态/变更游标）均落 DB，进程重启可续跑；内存只做临时缓存与批处理。
