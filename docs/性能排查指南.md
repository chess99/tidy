## 0. 适用范围

这份指南用于排查以下类型问题：

- **接口慢 / 偶发几十秒**（典型：用户输入触发多请求并发，导致排队/资源争用）
- **CPU/MPS/CUDA 占满**导致整体变慢
- **冷启动**（首次请求明显慢）与“羊群效应”（多个请求同时触发模型加载）

本仓库的核心观测手段是 **server 与 ai-service 的内置 profiling**：

- server：`/api/*` 支持 `?profile=1` 或 header `x-tidy-profile: 1`
- ai-service：`/clip/*` 支持 `?profile=1` 或 header `x-tidy-profile: 1`

> 建议：先用 profiling 把“慢在谁身上”定位清楚，再做交互/架构调整，避免盲调。

## 1. 快速定位套路（推荐顺序）

### 1.1 先确认“端到端慢”还是“排队慢”

对目标接口加 profiling（以智能搜索为例）：

```bash
curl -s 'http://localhost:5173/api/search?profile=1' \
  -H 'Content-Type: application/json' \
  --data-raw '{"query":"海","page":1,"limit":50,"topK":1000,"minScore":0.51}' \
| python -c 'import json,sys; j=json.load(sys.stdin); p=j.get("profile") or {}; print("server.totalMs", round(p.get("totalMs",0),2)); \
  print("server.eventLoopDelay", p.get("eventLoopDelay")); \
  [print(" -", s.get("name"), round(s.get("ms",0),2), s.get("extra")) for s in (p.get("steps") or []) if s.get("name") in ("ai.http","ai.json","clip.text.embed.done","clip.ann.done","db.files.batch","inflight")]; \
  end=(p.get("steps") or [])[-1].get("extra") if (p.get("steps") or []) else {}; ai=(end or {}).get("aiService") or {}; \
  print("ai.totalMs", round(ai.get("totalMs",0),2)); \
  [print("  *", s.get("name"), round(s.get("ms",0),2), s.get("extra")) for s in (ai.get("steps") or []) if s.get("name") in ("clip.slot","clip.get_encoder","clip.encode_text","clip.encode_text.model.encode_text")]'
```

**解读：**

- `server.steps.ai.http.ms` 大 → **主要耗时在 server→ai-service HTTP + 推理**
- `ai.profile.steps.clip.slot.extra.waitMs` 大 → **排队导致慢（并发超过 ai-service 的推理并发上限）**
- `server.eventLoopDelay.maxMs` 很大 → server 自己被 CPU/同步 IO 拖住（需看 server 的 cpuMs/DB）

### 1.2 复现“输入导致多请求并发”

典型场景：IME 拼写/联想输入会产生多个 query（`h/ha/hai/海...`），如果每次都触发 `/api/search`，会造成 ai-service 推理并发争用与长尾。

建议的交互策略（本仓库已采用）：

- **重操作（推理/向量检索）**：输入草稿 + 勾勾/Enter 手动应用，避免 IME 阶段连发
- 如仍需“自动触发”：必须加 **debounce + 取消 in-flight**（仅保留最后一次）

### 1.3 定位 ai-service 推理是否资源争用

ai-service 的 profiling 里重点看：

- `clip.encode_text.model.encode_text`：纯推理耗时（CPU/MPS/CUDA 真正干活）
- `clip.slot.waitMs`：排队等待（说明并发超过可承受）
- `rssMaxKb`：进程 RSS 峰值（是否内存压力导致抖动）

并发压测（6 并发，观察 waitMs 阶梯增长）：

```bash
node server/scripts/profile-clip-text-embed.js --text 海 --concurrency 6
```

### 1.4 定位 server 端是否有“自我阻塞”

server 的 profiling 提供：

- 每一步 `ms/cpuMs/rssDeltaKb/heapUsedDeltaKb`
- `eventLoopDelay.meanMs/maxMs`（事件环延迟）

当 `eventLoopDelay.maxMs` 很高时，优先检查：

- 是否有大块同步 CPU（排序/大数组操作）
- 是否有同步 IO 或大量 JSON 序列化
- DB 查询是否缺索引（可用 `EXPLAIN QUERY PLAN`）

## 2. 常见根因与对策（高频）

### 2.1 并发推理导致长尾（最常见）

现象：

- 单请求 profile 看起来 1–2s
- 用户体验却是 10s+ 或几十秒
- ai-service `clip.slot.waitMs` 变大、甚至出现错误

对策（优先级从高到低）：

- **前端：减少触发频率**（手动应用 / debounce）
- **前端：取消 in-flight**（AbortSignal/React Query signal）
- **ai-service：限制推理并发**（`TIDY_CLIP_CONCURRENCY=1` 起步；增大需看 waitMs 与 P95/P99）
- **ai-service：模型加载加锁**（避免冷启动羊群）

### 2.2 冷启动/模型加载

现象：

- 重启后第一条请求明显慢
- 多个请求同时来，全部被拖慢

对策：

- 服务启动后做 warmup（server 已做 best-effort warmup）
- ai-service 模型加载加锁（避免重复加载）

### 2.3 数据库/索引问题

现象：

- `db.*` 步骤耗时异常大
- `cpuMs` 不高但 `ms` 高：可能在 IO 等待

对策：

- 用 `EXPLAIN QUERY PLAN` 看扫描/索引使用情况
- 限制一次查询返回规模、避免 `IN (...)` 过大
