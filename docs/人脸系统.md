# 人脸系统（Face Pipeline）

本文档描述 Tidy 的人脸系统实现、数据模型、接口与脚本、调参方式。

> 目标：让"后续接手的 AI/开发者"能快速理解：**现在做了什么、怎么用、哪里可调、为什么这样设计**。

---

## 1. 系统功能

### 1.1 核心能力

- **检测并入库人脸**：对图片资产检测人脸框，并把每个脸的 embedding（descriptor）与 box 入库到 `faces` 表。
- **"有人/人数"筛选**：支持在"全部文件"页筛出有人脸的图片，以及限制"同图中不同人的人数范围"。
- **聚类生成 `people`**：支持通过 DBSCAN（cosine distance）把 `faces` 聚成 `people`，并回填 `faces.person_id`。
- **详情面板交互**：
  - 显示该图片中所有脸（裁剪小头像）。
  - 可把该人物加入左侧筛选（找该人照片、找合照）。
  - 支持 **merge/split** 做最小人工纠错。
- **维护脚本**：
  - 清空人脸记录、全量扫描、单张测试、聚类校准、重聚类。

### 1.2 已知限制

- **人脸检测误检仍可能存在**：已通过阈值+尺寸过滤大幅降低，但仍受模型能力与素材类型（截图/海报/卡通）影响。
- **聚类质量受阈值影响很大**：DBSCAN 的 `eps` 需要按你库的 embedding 分布校准。
- **重聚类目前会重建 `people`**（会丢失 name），后续将按"保留已命名 person（锚点对齐）"改造。

## 2. 数据模型（SQLite）

### 2.1 `faces`（人脸实例）

字段（关键）：
- `id`：PK
- `hash`：对应 `assets.hash`
- `person_id`：可空；聚类后指向 `people.id`
- `descriptor`：JSON 字符串（float 数组）
- `box`：JSON 字符串（`{x,y,width,height}`）
- `score`：检测置信度
- `created_at`

### 2.2 `people`（人物实体）

字段（关键）：
- `id`：PK
- `name`：可空（用户标注）
- `avatar_face_id`：可空；人物头像对应 `faces.id`
- `created_at/updated_at`

> 约定：UI 展示人物时，优先展示 `name`，没有 name 时展示 `ID:<id>`。

### 2.3 `assets` 补充字段

- `face_scanned_at`：人脸扫描标记（用于"补扫"时跳过已扫过的图片）

## 3. 检测入库流程

### 3.1 关键文件

- `server/src/scanner/face.js`
  - `detectFaces(imagePath)`：通过 `aiClient.detectFaces()` 调用 ai-service 的 `/detect+embed` 接口
  - `processImageFaces(imagePath, hash)`：写入 `faces` 表（当前版本**不自动造人**，`person_id=NULL`）
- `server/src/services/aiClient.js`
  - `detectFaces({ imagePath })`：读取图片文件，转换为 base64，调用 ai-service 的 `/detect+embed`，并转换输出格式
- `server/src/scanner/index.js`
  - `scanFaces()`：挑选需要扫描的图片资产，逐张调用 `processImageFaces`，并写 `assets.face_scanned_at`

### 3.2 模型与依赖

- 推理服务：`ai-service`（Python FastAPI）
- 模型库：`insightface==1.0.1`（使用 `FaceAnalysis(name="buffalo_l")`；Windows 通过 wheel 安装，不走本机编译）
- 推理引擎：ONNXRuntime（支持 CPU/GPU）
- 图像处理：OpenCV（在 ai-service 中）
- 模型文件：由 `insightface` 库自动下载和管理（首次运行时下载到 `~/.insightface/models/`）

### 3.3 关键可调参数（减少误检）

在 `server/src/config.js`（支持 env 覆盖）：

- `FACE_MIN_CONFIDENCE`：默认 `0.7`
  - 越大越严格（误检下降、漏检上升）
- `FACE_MIN_PX`：默认 `40`
  - 人脸框最小边长阈值（过滤小框假脸/远处小脸）

`detectFaces()` 还做了额外过滤：
- 过于极端的宽高比（明显异常框）会过滤掉

## 4. 聚类（DBSCAN）与重聚类

### 4.1 设计原则

- **入库阶段不做聚类**：避免"边扫边造人"导致早期错误固化。
- **聚类是显式操作**：可重跑、可调参、可回滚（这也是 PhotoPrism/Immich 常见模式）。

### 4.2 当前实现

关键文件：
- `server/src/services/faceClustering.js`
  - cosine distance
  - DBSCAN（O(n^2)，小中规模可用）
- `server/src/services/reclusterPeople.js`
  - 读取 `faces.descriptor`
  - DBSCAN 输出 cluster labels
  - 重建/回填 `people` 与 `faces.person_id`
  - 用 cluster 内最高 `score` 的 face 当 `avatar_face_id`

关键脚本：
- `server/scripts/recluster-people.js`
  - `--eps` 默认 `0.04`（cosine distance）
  - `--minSamples` 默认 `2`
- `server/scripts/cluster-calibrate.js`
  - 扫 eps 区间，打印：clusters、people、noise、平均簇大小
  - 用它来"找 eps 合理区间"

> 经验：InsightFace embedding 的 cosine distance 分布需要根据实际数据校准。建议先用 `cluster-calibrate.js` 扫描 eps 区间，找到适合你库的合理范围（通常可能在 `0.02~0.06` 之间，但需以实际数据为准）。

### 4.3 重要注意：重聚类会丢 name（待改）

当前 `reclusterPeople()` 采用"重建 people"策略（会 `DELETE FROM people`），因此 **会丢失 `people.name`**。

后续改造方向（下一步要做）：
- **保留已命名 person（锚点对齐）**：重聚类生成的簇尽量映射回已有 name 的 person，从而不丢名字。

## 5. API（HTTP）

### 5.1 `faces` 路由

文件：`server/src/routes/faces.js`（挂载：`/api/faces`）

- `POST /api/faces/scan`
  - 触发后台补扫（对 `face_scanned_at IS NULL` 的图片资产）
  - 若 AI 人脸能力当前不可用，任务会快速失败并返回 `blockedReason='faces_unavailable'`，不会继续逐张扫图
- `POST /api/faces/reset-scan-marker`
  - 重置 `assets.face_scanned_at = NULL`
  - 可选：`clearFaces/clearPeople`
- `POST /api/faces/recluster`
  - 触发 DBSCAN 重聚类（参数：`eps/minSamples`）
- `GET /api/faces/asset/:hash`
  - 返回某张图片的 faces（含 box/score/person_name）
- `GET /api/faces/people`
  - 返回 people 列表
- `PUT /api/faces/:id`
  - 手动把 face 绑定到 person（或解绑）
- `POST /api/faces/create-from-face`
  - 以某个 face 创建 person（用于人工标注）
- `POST /api/faces/people/:id/merge`
  - merge 两个人（把 from 的 faces 全移动到 target，删除 from）
- `POST /api/faces/people/:id/split`
  - split：把一组 faceIds 移到新建 person

### 5.2 系统状态路由

文件：`server/src/routes/system.js`（挂载：`/api/system`）

- `GET /api/system/status`
  - 返回归一化后的 AI capabilities，以及 `faces_scan` / `clip_enrich` 的最近任务摘要
  - 供简化设置页中的系统健康区域展示；手动任务入口仍保留在任务/维护相关界面

## 6. 前端交互点

### 6.1 入口（配置页）

文件：`client/src/components/ConfigView.jsx`

- 简化设置页展示紧凑的系统健康状态（AI 可用性 + 最近任务摘要）
- 人脸补扫（入库）、重置人脸标记（可选清空 faces/people）、重聚类（生成人物）等手动控制保留在任务/维护相关界面，不塞进简化设置页

### 6.2 详情面板（小头像+筛选）

文件：`client/src/components/AssetFacesPanel.jsx`

- 展示 faces 的裁剪小头像
- 显示 person（name 或 ID）
- 放大镜：把 person 加入左侧筛选
- merge/split：最小人工纠错

### 6.3 左侧筛选（"有人/人数"）

文件：`client/src/components/FilesFilters.jsx` + `server/src/routes/files.js`

- `hasPeople`：仅有人脸
- `personCountMin/personCountMax`：同图中不同人物数量范围
  - 当前口径：`COUNT(DISTINCT COALESCE(person_id, id))`
    - 如果某些 face 还未聚类（person_id NULL），也会被计入人数（用 face_id 兜底）

## 7. 脚本清单（server/scripts）

- `clear-faces.js`：清空 faces/people + 重置扫描标记
- `scan-faces.js`：全量扫图写 faces（`--all` 代表强制全量重扫）
- `scan-one-face.js`：单张图片检测（`--no-db` 不落库；`--reset` 落库前清理）
- `cluster-calibrate.js`：eps 扫描（建议先跑来定 eps）
- `recluster-people.js`：重聚类生成 people 并回填 faces.person_id

## 8. 运行/调试建议

### 8.1 快速跑通推荐流程

1) 确保 ai-service 运行：
- ai-service 会在首次运行时自动下载 InsightFace 模型
- 模型下载到 `~/.insightface/models/`（无需手动操作）

2) 入库（人脸补扫）：
- UI：配置页点"人脸补扫（入库）"
  - 或脚本：`npm run faces:scan` / `npm run faces:scan:all`

3) 校准 eps：
- `npm run faces:calibrate -- --from 0.02 --to 0.08 --step 0.01 --minSamples 2`

4) 重聚类：
- UI：配置页点"重聚类（生成人物）"
  - 或脚本：`npm run faces:recluster -- --eps 0.04 --minSamples 2`

### 8.2 误检太多怎么调

- 提高 `FACE_MIN_CONFIDENCE`（例如 0.75/0.85）
- 提高 `FACE_MIN_PX`（例如 60/80）
- 对截图/海报多的库：考虑只对"相机照片"或特定目录扫脸（后续可加）

## 9. ai-service 集成

### 9.1 服务架构

目录：`ai-service/`

- FastAPI 服务：
  - `GET /health`
  - `POST /detect+embed`（输入 base64，输出 box/kps/embedding/score）
- 使用 `insightface` 的 `FaceAnalysis(name="buffalo_l")`
- 依赖在 `ai-service/requirements.txt`
- **已与 Node server 打通**：通过 `server/src/services/aiClient.js` 的 `detectFaces()` 调用

### 9.2 集成方式

- Node 侧通过 `aiClient.detectFaces({ imagePath })` 调用 ai-service
- Node 侧也会先读取 `GET /health` 做能力预检；该接口只检查 import/依赖，不主动加载重模型
- 图片文件读取后转换为 base64，发送到 `/detect+embed`
- ai-service 返回的格式（`[x1,y1,x2,y2]`）自动转换为兼容格式（`{x,y,width,height}`）
- 配置：`AI_SERVICE_URL`（默认 `http://localhost:8002`）

### 9.3 能力预检与自动恢复

- `/health` 会返回 `capabilities.faces` / `capabilities.clip`，结构均为 `{ available, code, message }`；CLIP 可用时额外返回 `model`
- `faces_scan` 开跑前会先检查 `capabilities.faces`；不可用时直接以 `faces_unavailable` 失败，避免无意义全量扫描
- 当人脸能力从不可用恢复为可用后，server 会按节流周期自动检查是否存在未扫图片；若有则自动补入一个 `faces_scan missing` 任务
- 为避免抖动反复重试，最近一次自动恢复任务若刚失败，会进入短暂 backoff 再尝试

### 9.4 CPU/GPU 支持

- CPU：`onnxruntime` 默认即可
- NVIDIA GPU：使用 `onnxruntime-gpu` + CUDA runtime（在 Windows 台式机最适合）
- Ubuntu NAS 集显：通常走 CPU（除非走 OpenVINO 等专门路线）
- macOS：先 CPU；GPU（Metal/CoreML）属于后续增强

## 10. 待办清单（未来迭代建议）

1) **重聚类不丢 name**（优先级高）
   - 方案：保留已命名 person（锚点对齐）+ 只重分配未命名簇
2) **"以脸搜脸"**：基于 descriptor 做相似搜索（不依赖 person_id）
3) **更强解码与预处理**：用 `sharp` 统一转码，减少 `canvas` 对 webp/tiff 等失败
4) **更好的聚类**：HDBSCAN/层次聚类/定期重算 centroid
5) **可观测性**：人脸扫描进度/失败原因统计/样本导出
