# Tidy：关键设计决策与取舍（Decision Log）

> 目标：记录**长期稳定**的设计理念/决策/取舍，帮助后续迭代保持一致性；不记录已实现的细枝末节。

## 1) 产品边界：坚持“整理工具”，不是相册 App

- **定位**：核心是“文件去重 + 归档到真实文件夹 + 可恢复的一致性”，而不是做完整的人像/地点相册体系。
- **组织方式**：以“文件夹/归档（Folders/Archives）”为中心，允许你按旅程/主体/记录类等建立长期结构；更复杂的“相册特性”只做必要的扩展位（见标签）。

## 2) 核心实体：`assets(hash)` vs `files(path)`

- **内容实体**：`assets.hash` 是内容主键（同一内容只算一份）。
- **物理实例**：`files.path` 只是内容在磁盘上的一个副本路径（允许多份、允许缺 hash）。
- **关键规则**：所有“整理/删除/标注/归档”的语义都应绑定 `assets(hash)`，避免重复文件需要重复操作（也解决“删了又被同步回来”的复活问题）。

## 3) 真实落盘：整理操作立即执行文件系统变更

我们放弃“先虚拟归类，之后统一同步”的复杂模型，改为：

- **一次明确操作（添加到/删除/入Trash）** → **后端立即移动/入Trash** → DB 更新完成后再回给前端展示结果。
- **原因**：虚拟/落盘双态会带来极高复杂度（对账、冲突、崩溃恢复、跨进程一致性），与“整理工具”的定位不匹配。

## 4) 托管目录（Managed Root）与扫描边界

- **扫描根**：无默认扫描目录；用户自行添加扫描根（config.json `scanRoots`），Discover 仅扫描已启用的根。
- **托管归档目录**：工具整理后的目标目录 `MANAGED_ROOT` 由用户选择（默认 `~/Pictures/Tidy`，可通过设置界面配置）。
- **扫描可包含** `MANAGED_ROOT`：若某扫描根涵盖托管目录，其中文件即“已整理”的事实来源（用于 DB 丢失/损坏/人工整理后的恢复与对账）。
- **Trash**：工具回收站 `TRASH_DIR` 默认在 `DATA_DIR/trash`（可环境变量覆盖），只用于“用户删除内容”的**最后一份留底**（用于恢复/对账）。

### 4.1) "保留" vs "整理到目录" 的语义区分

- **"保留"（Keep/Sorted）**：原地标记状态为 `sorted`，表示用户确认要保留此内容，但**暂不移动**到目标目录。
  - 适用场景：用户想先筛选/去重，批量确认要保留的文件，之后再统一整理到文件夹。
  - 物理文件位置不变，仅状态变更。

- **"整理到目录"（Organize/Add to Folder）**：将文件**移动**到 `MANAGED_ROOT/<folder>/`，同时标记为 `sorted`。
  - 适用场景：真正的归档操作，文件被移动到托管目录。
  - 会去重（同 hash 只保留一份），副本直接删除。

- **关系**：两者最终都标记为 `sorted` 状态，但"保留"是原地标记，"整理"是移动归档。用户可灵活选择 workflow。

## 5) 去重策略（Dedup Policy）

- **判断依据**：以 `assets.hash` 判定重复。
- **整理时去重**：同 hash 多份物理文件 → **只保留一份**移动到目标文件夹，其余副本 **直接物理删除**（不进入回收站）。
- **目的**：让归档目录天然“内容唯一”，同时避免回收站被“去重残骸”污染（回收站只承载用户删除的留底）。

## 5.1) 重复项工具：hash vs 相似重复（pHash）

- **背景**：同一场景的不同分辨率/压缩版本在用户视角上属于“重复”，但内容 hash 不同。
- **策略**：
  - 完全重复：按 `assets.hash` 分组。
  - 相似重复：对图片计算 `pHash(64-bit)`，按 Hamming 距离阈值分组处理（默认中等阈值）。
  - 交互以“物理文件实例”为主；当某 hash 的所有实例都被选中删除时，升级为删除该 asset（保留最后一份到回收站）。

## 6) 一致性与可恢复：`file_ops` 操作日志 + `sync` 对账任务

为避免“文件系统操作成功但 DB 没记上 / DB 记了但文件没动”：

- **写操作日志表 `file_ops`**：记录 `move/trash/delete` 的 from/to/hash/状态/错误。
- **执行顺序（核心思想）**：
  - 写入 `file_ops(pending)`（可追溯）
  - 执行文件系统 move/trash（真实落盘）
  - 更新 DB（`files/path`、`album_assets`、`assets.status/target_path` 等）
  - 标记 `file_ops(done)` 或 `file_ops(error)`
- **`sync` 任务的语义**：从“同步变更按钮”演进为**恢复/对账入口**：
  - 重放 `pending file_ops`（崩溃恢复）
  - 兜底处理历史/遗留的 trash 标记，转成可追踪的 `file_ops`

## 7) “文件夹/归档（Albums）”的语义

- **Folders/Archives 不是虚拟合集**：它对应托管目录下的真实文件夹（`<MANAGED_ROOT>/<albumName>`）。
- **关系存储**：用 `albums` + `album_assets(album_id, hash)` 表示“内容属于哪个归档文件夹”，并可反查 `organized_to` 直接在 All Files 卡片上展示。

## 8) 标签能力：只做可扩展位（先手动，后模型）

- **设计目标**：支持“地点/主体/人物/食物/其他”的手动标签，并为未来识别模型预留扩展点。
- **绑定对象**：标签绑定在 `assets(hash)` 上（而不是 `files`），避免重复文件重复标注。
- **范围**：第一阶段只需要最小 CRUD + 关联表（`tags`/`asset_tags`），是否做“按标签筛选”可渐进实现。

## 9) UI/UX 取舍：All Files 专注“可整理性”，Folders/Archives 专注“归档浏览”

### All Files（`FilesGrid`）

- **必须直接可见的信息**（无需点开）：
  - 未整理：文件名 + 物理路径
  - 已整理：整理去向 `organized_to`
  - 重复：`dup_count` 标记
- **固定尺寸**：卡片/行高固定，避免加载抖动。
- **多选与批量操作**：筛选后多选 → “添加到(选择/新建文件夹)”或“删除入Trash”等。
- **快捷筛选**：未整理、重复、日期范围、路径包含、hash 精确（用于定位同内容）。

### Folders/Archives（原 Gallery/AlbumsView）

- **展示真实归档文件夹列表** → 点进查看内容。
- **目标**：快速确认“归档结构”与“某归档包含哪些内容”。

## 10) 滚动与日期定位：date-index + overlay

- **问题**：大库滚动时不能靠“顺序加载分页”，需要随机访问可视区域页；日期提示也必须稳定不跳动。
- **方案**：
  - 后端提供 `GET /files/date-index`：生成 month/day 的 index→日期分界点。
  - 前端 overlay 在滚动时显示当前区域日期，并支持“跳到某月”。
  - 拖动滚动条时请求节流/降采样，停止后补齐 overscan。

## 11) 增量更新：SSE + batch patch（避免整列表刷新）

- **原则**：扫描/哈希/缩略图补全是渐进的，UI 不应整页刷新。
- **方案**：后端写 `changes` 表并用 SSE 推送 `{cursor, files[], assets[]}`；前端用 batch API 拉取最新行并只 patch 已加载项。

## 12) Windows/工具链现实约束：UI 组件尽量用 shadcn CLI 生成

- **原因**：Windows 上 dev 进程容易锁 `esbuild.exe`，而 shadcn 的本质是“把组件源码落到 repo 里”；我们倾向于用 CLI 生成后把源码纳入版本控制，避免手工维护大量细节。
- **原则**：优先使用 shadcn 的“配方/组件生成”，而不是手写 day-picker/popover 等细节类 UI。

## 13) 脏数据与维护：大小写路径重复、缺失文件

- **路径大小写策略**：保持现状（大小写敏感），不强制全局归一（避免对用户真实路径施加隐藏变更）。
- **提供清理工具**：通过脚本（如 `server/scripts/repair-db.js`）提供：
  - `--report`：报告仅大小写不同的重复路径
  - `--dedupe-case`：按策略清理 DB 行（是否动磁盘应明确选择）
  - 缺失文件标记/清理（report + 可选 fix）
- **重扫（删库再扫）**：
  - hash 必然重算（内容主键必须可靠）
  - 缩略图应尽量复用：若 `thumbnails/<hash>.jpg` 已存在则跳过生成并回填 DB 状态

## 14) 文档策略（持续维护）

- **把“为什么”写进 `决策记录`**：定位、边界、一致性与取舍。
- **把“怎么做”写进 `设计文档`**：架构、数据模型、工作流与 API 约定。
- **避免实现细节污染**：具体 UI 像素/组件文件清单、某次修 bug 的过程不写进 docs（除非它改变了长期决策）。
