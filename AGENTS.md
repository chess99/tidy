# Agent Instructions

## 工作流

- 除非用户主动要求切分支，否则一般都直接在主分支或当前需求分支进行开发。
- 每完成一个完整迭代（实现、验证、必要文档更新都结束）后，主动提交本次自己修改的文件，不等用户额外要求。
- 提交时只包含本次自己修改的内容，禁止提交其他 agent 或用户并行修改的文件。
- 提交前用 `git diff --staged` 确认暂存区只包含本次变更。

## 项目定位

- Tidy 是内部优先项目，默认不为外部用户、第三方集成或历史调用方保留额外兼容包袱。
- 长期可维护性、清晰性、性能与一致性优先于临时交付捷径。
- 不留下 TODO/FIXME、临时 hack、弃用路径、双写逻辑、兼容适配器或“以后再改”的技术债。

## 代码决策

- 优先沿用现有架构、命名、目录结构、数据流和交互模式；除非收益明确且足够大，不引入异类写法。
- 能通过删除、收敛边界或重构降低复杂度时，不用叠加逻辑做补丁式修复。
- 接口、数据结构、路由、组件或模块边界不合理时，可以破坏性重构为更合理的形态，但必须同步更新所有调用点。
- 抽象要小而窄：当同时修改多个重复点，或相同逻辑短期会频繁变化且容易漏同步时再抽象；单点逻辑可以保留局部重复。
- 只为已知需求或极近期演进预留接口，避免为了“可能会用”提前增加复杂度。
- 非直觉或有争议的实现必须用简短注释说明原因，注释解释“为什么/约束”，不要解释代码正在做什么。

## 质量要求

- 处理边界条件、空值、异常路径、并发和竞态；禁止静默吞错，除非有明确理由并注明。
- 入参、出参和状态类型要清晰；可选字段优先用 `undefined` 表示“未提供”；严格限制 `any`。
- 将 I/O、网络请求、存储、数据库等副作用收敛到边界层，核心业务逻辑尽量保持可局部验证。
- 用户可见错误要可操作；日志包含关键上下文，但不能打印 token、密钥或用户隐私数据。
- UI 的 loading、禁用、空态、错误态、数据回显、分页和搜索行为要与项目内相似场景一致。
- 避免无意义请求和重渲染；列表、搜索等高频输入按项目既有模式做 debounce、缓存或预加载。

## 变更收尾

- 改了接口、函数签名、返回结构或持久化数据结构，必须同步更新调用方、文档和迁移/修复逻辑。
- 涉及 DB、文件或缓存结构变更时，在仓库内提供可复现的一次性迁移脚本或自动修复逻辑。
- 删除冗余代码、死代码、过时配置和无用开关；不要保留“以防万一”的分支。
- 文档只更新与本次改动直接相关且必要可发现的层级；不要为了形式做全量文档铺开。
- 完成前运行与改动风险匹配的验证命令，并在无法验证时说明原因。
